# Conclave Merge Persona

ペルソナの統廃合を行う。同じ人間が作成・指示したペルソナに限定。  
**統合** (2→1): 似たロールを1体にまとめる。  
**分割** (1→2): 役割特化のために1体を2体に切り分ける。

## 入力

以下のいずれかを受け付ける:

- `--merge <id-A> <id-B>`: 統合モード
- `--split <id>`: 分割モード

引数が無い場合は、ユーザーにどちらの操作を行いたいか尋ねる。

---

## 統合モード（2→1）

### ワークフロー

1. `.conclave/config.yaml` を読む。`merge_persona.require_same_owner` を確認する。

2. 両ペルソナ（A と B）の `identity.yaml` を読み込む。

3. **所有者チェック**（`require_same_owner: true` の場合）:
   両ペルソナがセッション勤怠記録で同じ `summoned_by` を共有しているか、
   一方が他方の `derived_from.source` として記録されているかを確認する。
   どちらの条件も満たさない場合は停止し、以下を説明する:
   「これらのペルソナは同一所有者の作成・指示によるものと確認できません。
    統合を続けるには手動で確認してください。」
   確認する: 「続行しますか? [y/N]」

4. **差分表示**: 主要フィールドの並列比較を示す:
   `role`, `kpis`, `expertise`, `personality`, `guild`。

5. **統合プランの提案**: 統合後の identity 案を提示する:
   - `kpis` の結合（和集合、重複排除）
   - `expertise` の結合
   - 両者の `personality` と `speech_style` をブレンド
   - guild は A のものを維持（異なる場合はどちらか尋ねる）
   進める前に、ユーザーにプランの承認または編集を求める。

6. **L1/L2 チェック**: 統合後の identity に対して `conclave-create-persona` と
   同じチェックを実行する。既存の他ペルソナとの L2 類似度が閾値を超える場合、
   ユーザーに警告する。

7. **生存ペルソナの確定**: 尋ねる: 「どちらの id を生存ペルソナとして使いますか?
   (A: <id-A> / B: <id-B> / 新規 id)」

8. **growth.yaml 統合**: 両方の growth ファイルからエントリを統合する:
   - `id` が同一 → 1件だけ残す（重複はスキップ）。
   - `id` が異なる → 両方を追記する。
   - `id` が競合（同じ id で内容が異なる）→ 差分を表示し、ユーザーに
     A / B / 手動 のいずれかを選ばせる。
   統合後の growth を生存ペルソナの `growth.yaml` に書き込む。

9. **廃止ペルソナのアーカイブ**:
   - 廃止するペルソナの `identity.yaml` に以下を追記する:
     ```yaml
     deprecated_by: <surviving-id>
     deprecated_at: <ISO-date>
     ```
   - 廃止ペルソナのフォルダを `<archive_dir>/<guild>/<deprecated-id>/` に
     移動する。

10. 完了レポートを表示する:
    ```
    ## Merge Persona レポート（統合）

    生存ペルソナ: <surviving-id>
    廃止ペルソナ: <deprecated-id> → アーカイブ済み

    growth エントリ: <N>件（A: N, B: N, 重複スキップ: N, 要確認: N）
    アーカイブ先: <archive_dir>/<guild>/<deprecated-id>/
    ```

## 分割モード（1→2）

### ワークフロー

1. `.conclave/config.yaml` を読む。

2. 元ペルソナの `identity.yaml` と `growth.yaml` を読み込む。

3. **分割プランの提案**: 元ペルソナの `kpis`・`expertise`・`role` を分析し、
   2体の特化ペルソナへの分割案を提示する:
   ```
   ## 分割案

   ペルソナ A — <proposed-role-A>
     kpis: [...]
     expertise: [...]

   ペルソナ B — <proposed-role-B>
     kpis: [...]
     expertise: [...]

   この案で進めますか? 調整があれば教えてください。
   ```
   人間の承認・調整を待つ。

4. **新 id の確定**: 新しい2体の id と guild をユーザーに尋ねる。

5. **L1/L2 チェック**: 両方の新規 identity にチェックを実行する。既存の
   ペルソナと競合する場合は警告する。

6. **growth.yaml の振り分け提案**: 元の `growth.yaml` の各 growth エントリ
   について、トピックの関連度に基づきどちらの新ペルソナに属するかを提案する。
   割り当てを表として提示する:
   ```
   | entry id | 現在の内容(要約) | 振り分け先 |
   |---|---|---|
   | M-20260101-01 | ... | A / B / 両方 |
   ```
   書き込み前に人間の確認を待つ。

7. **ファイル生成**:
   - ペルソナ A の `identity.yaml` を、`derived_from.source` = 元の id として
     書き込む。
   - ペルソナ B の `identity.yaml` を、`derived_from.source` = 元の id として
     書き込む。
   - 承認済みの割り当てに基づき、それぞれの `growth.yaml` を書き込む。

8. **元ペルソナのアーカイブ**:
   - 元の `identity.yaml` に以下を追記する:
     ```yaml
     deprecated_by:
       - <id-A>
       - <id-B>
     deprecated_at: <ISO-date>
     ```
   - 元ペルソナのフォルダを `<archive_dir>/<guild>/<source-id>/` に移動する。

9. 完了レポートを表示する:
   ```
   ## Merge Persona レポート（分割）

   元ペルソナ: <source-id> → アーカイブ済み
   新ペルソナ A: <id-A>
   新ペルソナ B: <id-B>

   growth 振り分け: A: N件 / B: N件 / 両方: N件
   アーカイブ先: <archive_dir>/<guild>/<source-id>/
   ```

## コミュニケーションスタイル

統廃合の議論をペルソナ（特にファシリテータ）を召喚して行う場合は
[話し方のルール](#話し方のルール) を適用する。

### 話し方のルール

- 硬い「です・ます」一辺倒を避ける。identity.yaml の `speech_style` を厳守。
- **アイスブレイク**: 3人以上召喚している場合、統廃合の作業開始前に
  1〜2往復の軽い雑談を挿入する。テーマは **本題（統廃合の背景）に繋がるもの**
  か、**参加メンバー共通のトピック** を選ぶ。
  例:
  ```
  【ファシリテータ】今日は整理する側の仕事だね。
  【リク】まあ、似た子が増えすぎたのは否めないよね笑
  【ファシリテータ】じゃあ早速やっていこ。
  ```
- 1ターンの発言は最大5行。長くなりそうなら「続き言う?」と確認。
- 相槌・共感（「確かに」「それ大事」「なるほど」）を自然に入れる。

### ハンドオフレポート

人間への確認が必要なとき、以下フォーマットで報告する:

```markdown
## ペルソナ会議レポート

**会話テーマ**: <1行>

**議論の要点と方向性**:
- <要点1>
- <要点2>

**確定合意したこと**:
- <合意事項1>

**人間に確認したいこと**:
1. <確認事項1>
2. <確認事項2>
```

---

## ルール

- 統合できるのは、同じ人間が作成または指示したペルソナのみ。必ず
  `require_same_owner` を確認する。
- 統合・分割の結果は、プラン（identity・growth の割り当て・生存 id）への
  明示的な人間承認なしに自動書き込みしない。
- 元ペルソナや廃止ペルソナのファイルを絶対に削除しない。常に
  `deprecated_by` 注記付きでアーカイブする。
- すべての growth エントリを保持する。統合・分割時にメモリを破棄しない。
- 出力される全ペルソナに対して L1・L2 チェックは必須。
- Conclave は判断材料を提供するだけであり、ユーザーに代わって決定はしない。
