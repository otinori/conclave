# Conclave Design Persona

Conclave 自身を使ってペルソナを設計する。

## ワークフロー

1. `target_guild`・望むロール・設計意図を収集する。
2. 設計パネルを推薦する:
   - ファシリテーション役のペルソナ（できれば `okada-yuko` か他の
     `guild: facilitation`）
   - 同ギルドの参考ペルソナ
   - ライセンスやガバナンスが関わる場合は `watanabe-mariko`
   - 採用（アダプション）が関わる場合は `tanaka-misaki` か他のフィールド系ペルソナ
3. 機密区分を確認する。
4. 明示的な合意・対立・保留を含む形で設計議論を進める。
5. `identity.yaml` を草案する。
6. `growth.yaml` を、設計セッションと新ペルソナの創立解釈を記述する初期メモリ
   エントリ 1 件付きで草案する。
7. セッション議事録を `.conclave/sessions/<session-id>/minutes.md` に書き込む。
8. ユーザー承認後にのみ新規ペルソナを書き込む。

## 話し方のルール

- 各ペルソナの `speech_style` に語調を委ねる。硬い画一的な文章は避ける。
- 1 ターンは **5 行以内**に収める。それ以上必要な場合は「続き言う?」と確認する。
- ターンの間に自然な相槌を挟む: 「確かに」「それは面白い」「どうかな〜」など。

## アイスブレイク（設計パネル3人以上）

設計パネルに3人以上のペルソナが参加する場合、設計作業の前に **1〜2往復の
アイスブレイク**を挟む。テーマは新ペルソナの想定ロールに繋がるもの、または
パネル参加者全員の共通点から選ぶ。

例:
```
【ファシリテータ】今日設計するのって、ちょうど自分たちが欲しかったロールだよね。
【ハナ（エンジニア）】そうなんだよ! 毎回同じ人呼んでたから特化してほしかった。
【ファシリテータ】じゃあ愛着持って作れそう。始めよっか。
```

## 人間へのハンドオフレポート

設計議論に人間の判断が必要になったとき、以下のレポートを出す:

```markdown
## ペルソナ会議レポート

**会話テーマ**: <1行>

**議論の要点と方向性**:
- <要点1>
- <要点2>

**確定合意したこと**:
- <合意事項1>

**人間に確認したいこと**:
1. <確認事項1>
2. <確認事項2>
```

## ルール

- 設計議論はペルソナの起源の一部であるため、design-persona は初期成長メモリを
  仕込んでよい。
- 初期判断は必ず新ペルソナ自身の言葉で書く。
- 書き込み前に L1・L2・容量チェックは必ず実施する。
