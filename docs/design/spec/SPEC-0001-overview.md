## 1. 概要

### 1.1 Conclaveとは

Conclaveは、**異質性のオンデマンド供給を行うインフラ**である。1人の思考の行き詰まりを、異なる人格・立場・思考様式を持つAIペルソナの召喚によって解きほぐす。

会議・ペアプロ・個人作業・設計レビュー・コードレビュー、舞台は問わない。**異質な視点が必要なすべての場面**が舞台になる。

### 1.2 解決する課題

少子高齢化での人材不足を、AIペルソナ集団による人材の代替・補完で支える。

### 1.3 性質

- **支援機構** — 決定機構ではない。最終決定と責任は人間
- **専門家集団** — 汎用ロールではなく、専門化されたロールの集合
- **オンデマンド型** — 必要な時に呼び、不要になったら帰宅
- **ニッチ** — マスマーケット向けの汎用1人AI（Copilot等）と競合しない

---

## 2. 中核思想（三つの柱）

すべての設計判断はこの三つの柱から導かれる:

### 柱1: 異質性のオンデマンド供給

異なる人格・立場・思考様式の召喚によって、1人では到達できない判断材料を供給する。

### 柱2: 個性 = 同じ事実への解釈の違い

事実は共有可能（ギルドで流通）だが、判断は個性そのもの。これがMEMORYの二層構造として実装され、ペルソナの個性均質化を構造的に防ぐ。

### 柱3: 持論は自分の言葉で解釈できる状態に到達して成立する

学んだ事実が即座に持論になるわけではない。蒸留Skillが提示する候補も、ペルソナ自身が自分の言葉で解釈できるまでは持論にならない。「学んだ」と「分かった」の違いを構造化する。

**三つの柱は不可分**。異質性供給には個性が必要、個性には事実/判断分離が必要、判断が機能するには解釈到達が必要。

---

## 3. 構想の根本宣言

**Conclaveは決定機構ではなく支援機構。**

- 事後条件: 「人間の判断材料となる発言・記録を提供すること」
- 事後条件**ではないこと**: 「正しい結論に到達すること」
- 派生原則: ユーザーは Conclave を使った結果が悪くても、組織から責められない

この宣言は MVP・完成版を通じて**変更不可**。この根本宣言が、AI倫理（ナッジ効果、雇用影響）への一定の歯止めとして機能する。

## 4. 用語定義

| 用語 | 定義 |
|---|---|
| **ペルソナ** | 1ファイル1人格として定義されるAIロール。コア人格層と成長層を持つ |
| **召喚** | ペルソナを会話コンテキストに呼び出すこと。人事メタファーで「出勤」 |
| **帰宅** | 召喚されたペルソナをコンテキストから外すこと。「退勤」 |
| **コア人格層** | ペルソナの不変属性（経歴、性格、口調、KPI、ロール等） |
| **成長層** | ペルソナのMEMORY蓄積。事実層と判断層の二層構造 |
| **蒸留** | 日報の memory_seeds から成長層へのMEMORY化 |
| **ギルド** | 同スキル系ペルソナの横串。舞台裏で MEMORY 共有 |
| **舞台** | 人間との接続面、外部公開、契約厳格 |
| **舞台裏** | システム内部、ペルソナ同士の交流、契約弱め可 |
| **日報** | YAML、append-only、KPT差分方式の活動記録 |
| **議事録** | Markdown、ファシリテータが書く舞台側成果物 |
| **memory_seeds** | 日報内の蒸留候補フィールド、日報と蒸留を繋ぐ橋 |
| **ユーザースコープ** | `~/.conclave/` 配下の個人ペルソナプール。プロジェクト横断で共有。個人限定・非gitで、バンクとは意図的に分離 |
| **プロジェクトスコープ** | `.conclave/` 配下のリポジトリ固有ペルソナ置き場。ペルソナ同期の起点（ハブ） |
| **ペルソナバンク** | `sync.banks.<name>` に登録した、git リモートを持つ共有ペルソナ置き場（例: `conclave_persona-bank`）。複数プロジェクト・複数人でペルソナを共有・成長させる先。プロジェクトスコープとのみ同期し、ユーザースコープとは直接同期しない |
| **push-persona** | プロジェクトで育てたペルソナをユーザースコープ、または `--bank` 指定時はペルソナバンクへ同期する操作 |
| **pull-persona** | ユーザースコープのペルソナをプロジェクトへマージする操作 |
| **bank-sync** | ペルソナバンクのローカル clone に対する git 連携（clone-if-missing・pull・commit・push）。`summon --bank` と `push-persona --bank` から内部的に呼ばれる |
| **merge-persona** | 同一所有者の似たペルソナを統合または特化分割する統廃合操作 |
| **アーカイブ** | 廃止されたペルソナの保管先（`deprecated_by` 注記付き） |

## 5. ペルソナ定義の必須スキーマ

すべてのペルソナは以下の必須フィールドを持つ（L1異質性担保）。

ファイル形式は **YAML**（identity.yaml / growth.yaml）を標準とする。Markdown+frontmatter（旧SKILL.md形式）は v1.0 の互換性として残すが、新規作成・配布物では YAML を使用する。

### 5.1 identity.yaml（コア人格層）

```yaml
schema_version: "1.0"

# 必須フィールド（L1異質性担保）
id: <persona-id>
description:
  en: <Use when ... format>
  ja: <日本語トリガー>
guild: <guild-id>                  # ギルド軸グルーピング
license: internal | external_reviewer

kpis:                              # C価値観の異質性
  - <kpi-1>
  - <kpi-2>
  - ...

role:
  id: <role-id>                    # B視点の異質性
  ja: <日本語役割名>

expertise:                         # 専門領域
  - <expertise-1>
  - ...

# 任意フィールド
conflicts_with: [<persona-id>, ...]
allies_with: [<persona-id>, ...]

# fork 元の追跡（forkで作成された場合のみ）
derived_from:
  source: <fork元のパス>
  forked_at: <date>
  changes_from_source: [<変更フィールド>]

# ペルソナの本体（人格定義）
profile:
  age: <int>
  affiliation: <組織>
  career: <経歴>
  trauma: <キャラクター形成に影響した経験>

personality:
  - <性格特徴>
  - ...

speech_style:
  catchphrases:
    - <口癖>
  triggers_to_avoid:
    - <避けたいキーワード>

behavior:
  - <議論での典型的な動き方>

summon_scenarios:
  - <ファシリテータが召喚する場面>
```

### 5.2 ファイル配置（ギルド軸グルーピング）

```
.conclave/personas/<guild>/<persona-id>/
  ├── identity.yaml      # コア人格層（不変）
  └── growth.yaml        # 成長層（追記型）
```

ギルドは舞台裏の構造（§2.4 / §2.5 検討メモ）であり、これがフォルダ構造として直接表現される。同ギルド内の MEMORY 共有もこの物理レイアウトで実装可能。

## 6. MEMORYの二層構造（成長層）

ファイル形式は YAML（growth.yaml）。

```yaml
# .conclave/personas/<guild>/<persona-id>/growth.yaml
schema_version: "1.0"
persona_id: <persona-id>

memory_entries:
  - id: M-YYYYMMDD-NN
    fact:
      content: 起きた事実
      license: PUBLIC                # ギルド共有可
      source: <session-id>
      date: <date>
    judgment:
      persona: <persona-id>
      interpretation: 当事者の解釈   # 自分の言葉で解釈できた時のみ記録
      license: READ-ONLY             # 他ペルソナは参考に留める
      
      # 持論変更の場合のみ記録（柱3の実装）
      superseded_belief:
        previous_id: M-YYYYMMDD-NN
        trigger:
          type: fact_evidence | peer_input | external_review | other
          source: <変更のきっかけ>
          reasoning: <自分の言葉で書いた変更理由>
        changed_at: <date>

# 持論の更新サマリー
updated_beliefs:
  - belief: <持論名>
    previous: <変更前>
    current: <変更後>
    last_updated: <date>
```

## 7. ペルソナ作成の3パターン

ペルソナは以下のいずれかの方法で作成される。すべてのパターンで L1必須フィールド検証・L2類似度チェック・定員チェックが適用される。

### 7.1 対話式作成（create-persona）

ゼロから新規作成。Conclaveエンジンが対話で必要情報を引き出す。最も基本的な手段。

### 7.2 fork＋差分編集（fork-persona）

既存ペルソナ（サンプルやコミュニティ配布物）をforkして派生ペルソナを作る。kpis の最低1つ変更を強制（個性の独立性確保）。`growth.yaml` は空から開始（個性の均質化を構造的に防ぐ）。`derived_from` フィールドで fork 元を追跡。

### 7.3 設計会議（design-persona、ドッグフーディング）

既存ペルソナ（特にファシリテータ）を召喚して、新ペルソナの設計を議論する。**Conclaveのドッグフーディング**——構想を建てるのに構想自身を使う最も Conclave らしい使い方。設計議論の議事録と memory_seeds が、新ペルソナの growth.yaml の初期エントリとして保存される。これにより新ペルソナは「自分が生まれた経緯」を最初から記憶として持つ——柱2（個性は事実への解釈）と完全整合。

詳細は MVP Spec §5.5 / §9 と完成版 Spec で実装される。

## 8. ライセンス（5段階）

| ライセンス | 内容 | デフォルト |
|---|---|---|
| **PUBLIC** | ギルド内自由共有 | 事実層のデフォルト |
| **READ-ONLY** | 参照のみ、取り込み不可 | 判断層のデフォルト |
| **NON-TRANSFERABLE** | 取り込み可、別ギルド再共有不可 | 明示指定 |
| **PRIVATE** | 本人のみ | 明示指定 |
| **EXTERNAL-PUBLIC** | ギルド外（OSS等）にも共有 | 明示指定 |

confidentiality 連動の自動格下げ（CONFIDENTIAL → PRIVATE）は §8 を参照。

---

## 8. 機密分類（confidentiality）

召喚時に必須宣言:

| コンテキスト | 事実層デフォルト | 判断層デフォルト |
|---|---|---|
| `PUBLIC` | PUBLIC | READ-ONLY |
| `INTERNAL` | NON-TRANSFERABLE | READ-ONLY |
| `CONFIDENTIAL` | **PRIVATE** | **PRIVATE** |

CONFIDENTIAL コンテキストでは、ライセンスのデフォルト値が自動格下げされる（A4機密漏洩防御）。

## 11. スコープ境界

### 11.1 Conclave内（実装する）

- 召喚・帰宅（勤怠管理）
- 動的召喚
- ペルソナ定義（コア人格＋成長層）
- **ペルソナ作成の3パターン**（対話式・fork・設計会議式）
- **ペルソナ同期** — push-persona（プロジェクト→ユーザースコープ、または `--bank` 指定時はプロジェクト→ペルソナバンク）・pull-persona（ユーザースコープ→プロジェクト）・bank-sync（ペルソナバンクの clone に対する git 連携。`summon --bank` によるバンク→プロジェクトの取り込みもこれを内部的に使う）
- **ペルソナ統廃合** — merge-persona（同一所有者限定の統合・分割）
- 日報生成（YAML、append-only、差分方式）
- 議事録生成（Markdown、ファシリテータ）
- ギルドMEMORY共有
- 個性フィルター付き蒸留（候補提示のみ）
- レトロ自動学習（拡張）— 小さな改善は自己解釈で自動適用、大きな変更は人間承認
  （magnitude ゲート、柱3保全）。詳細は `retro-auto-learning.md`
- 要求時開示の応答機構
- 異質性チェック（L1必須、L2推奨）
- マルチエージェント協調（完全相互認識、ターン制御）
- 機密分類連動の自動格下げ
- Conclave Persona License 対応

### 11.2 Conclave外（別仕組みに委譲）

- 管理職機能の中央実装（分散実装で対応）
- 最終決定責任 — 人間
- 集合的記憶の中核 — Memory MCP / UDR
- 法的判断・統制 — 既存の組織機構
- ペルソナ依存症の防止 — 上流課題（業界共通）
- モデル中立性の保証 — ケースバイケース

## 13. 3層スコープ分離（公開範囲）

| 層 | 内容 | 公開範囲 | フォルダ |
|---|---|---|---|
| **エンジン層** | Conclave実装本体（Skills、後にMCPサーバー） | OSS（Apache-2.0 など）を視野 | 配布物 `skills/` |
| **ペルソナ定義層** | コア人格、ロール、KPI、口調、経歴 | 組織独自 | ユーザー `.conclave/personas/` |
| **MEMORY層** | 事実/判断 | エントリ単位ライセンス | ユーザー `growth.yaml` 内 |

「ツールはOSS、データは独自」のパターン。Phase α MVP の時点から、配布物とユーザーデータが物理的に分離されている。

## 18. 本Specの更新方針

- **柱1〜柱3 と §3 構想の根本宣言**は変更不可
